# Probe P4 — quack-jdbc `DatabaseMetaData` fidelity

**Date:** 2026-07-19
**Driver:** `com.gizmodata:quack-jdbc:0.2.0-alpha.4` (`driverName=quack-jdbc`, `driverVersion=0.1`)
**Server:** DuckDB (via Quack) `v1.5.4`, core `quack` extension, testcontainers (`debian:trixie-slim`)
**Test:** `TestQuackJdbcMetadataProbe` (this module). Raw transcript:
`REPORT-quack-jdbc-metadata-probe.transcript.txt` (regenerated by the test).

The probe creates one table (`all_types`) covering DuckDB's type surface + a NOT NULL column + a
unicode-named column, then records `DatabaseMetaData.getCatalogs/getSchemas/getTables/
getTableTypes/getColumns` and compares per-column against `duckdb_columns()` ground truth.

---

## Headline finding (load-bearing for the type map)

> **`TYPE_NAME` is 100% faithful; `DATA_TYPE` (the JDBC int code) is lossy.** For *every* type in
> the surface — including the ones the driver can't classify — `getColumns().TYPE_NAME` returns the
> **exact DuckDB type spelling** (`HUGEINT`, `UBIGINT`, `TIMESTAMP_S`, `INTEGER[]`,
> `STRUCT(a INTEGER, b VARCHAR)`, `MAP(VARCHAR, INTEGER)`, `ENUM(...)`, `UUID`, `INTERVAL`, `JSON`,
> `DECIMAL(38,10)`, …). Meanwhile `DATA_TYPE` collapses many of them to `OTHER(1111)`.

**Decision: the Doris type map keys off `TYPE_NAME` (a parsed DuckDB type string), NOT `DATA_TYPE`.**
This is the same shape doris-ducklake uses (it maps from the DuckLake `column_type` *string*), so
`DuckDbToDorisTypeMapper` is a string→`ConnectorType` mapper and never trusts the JDBC int code.
No fallback query to `duckdb_columns()` is needed for typing — `TYPE_NAME` already carries the
truth. (`duckdb_columns()` remains available as an oracle in tests.)

## getCatalogs / getSchemas / getTables

- **Catalogs:** `memory` (the user's in-process DB), `system`, `temp`. The user's tables live in
  **catalog `memory`, schema `main`**. → v1 exposes the connection's **main catalog only**
  (`memory` for an in-memory server; the attached DB name for a file/attached DB). `system` and
  `temp` are DuckDB-internal and MUST be excluded. Multi-catalog (`ATTACH`) is deferred — see
  *Open question: ATTACH*.
- **Schemas** (getSchemas): `memory.main`, plus `system.{information_schema,main,pg_catalog}`,
  `temp.main`. → filter to the main catalog's non-system schemas. In DuckDB a user's tables are
  in `main` unless they `CREATE SCHEMA`; we list whatever schemas exist under the main catalog.
- **getTables TABLE_TYPE:** base tables report **`TABLE`**; every system object reports
  **`SYSTEM VIEW`**. `getTableTypes()` = `{TABLE, VIEW, SYSTEM VIEW, LOCAL TEMPORARY}`. → list
  `TABLE` (and later `VIEW`); exclude `SYSTEM VIEW` / `LOCAL TEMPORARY`. This matches the Trino
  side's learning that DuckDB-JDBC reports base tables as `"TABLE"` — quack-jdbc agrees.
- **Identifier quote:** `"` (double-quote). Unicode identifiers round-trip: the `c_ünïcode`
  column came back verbatim.
- `supportsCatalogsInDataManipulation=true`, `supportsSchemasInDataManipulation=true`.

**Catalog/schema flattening decision (v1):** DuckDB has a 3-level namespace
(`catalog.schema.table`); Doris catalogs are 2-level (`database.table`). We flatten
**DuckDB schema → Doris database**, scoped to the connection's **main catalog only**. So
`listDatabaseNames` returns the main catalog's non-system schema names (typically just `main`),
and `getTableHandle(database, table)` resolves `main_catalog.database.table`. The main catalog
name is discovered from the connection (JDBC `Connection.getCatalog()` / the sole non-system
catalog), never hard-coded to `memory`.

Notes:
- **Unsigned promotion** follows doris-ducklake's rule: promote to the next signed type that holds
  the full unsigned range (`UTINYINT`→`SMALLINT`, `USMALLINT`→`INT`, `UINTEGER`→`BIGINT`). `UBIGINT`
  (0…2^64-1) does not fit `BIGINT`; it fits `LARGEINT` (int128) so we map it there (doris-ducklake
  used `DECIMALV3(20,0)` because DuckLake had no int128-decoding handler — we DO, via
  `DuckDbTypeHandler`, so `LARGEINT` is cleaner and the handler already normalizes wide unsigned to
  `BigInteger`). `UHUGEINT` (0…2^128-1) exceeds int128's positive range → **fail loud** (no
  correct Doris fit; never silently truncate).
- **BLOB → `VARBINARY`**: the BE `DuckDbTypeHandler` (doris-patches/be) decodes `BLOB`→`byte[]` into
  a `VARBINARY` column. doris-ducklake declares `blob` as `VARBINARY(65535)`; we follow it. (Without
  our handler we'd have to declare BLOB as STRING/hex — but we ship the handler, so `VARBINARY`.)
- **UUID → `STRING`** in v1: the `DuckDbTypeHandler` has no UUID arm yet (it's not in the plan's v1
  coverage table), and quack-jdbc returns UUID as a string over the wire. doris-ducklake maps
  `uuid`→`VARBINARY(16)` because its BE reads UUID from parquet as 16 raw bytes; our path is
  different (a JDBC result-set string), so `STRING` is the truthful v1 mapping. Revisit when the
  handler grows a UUID arm.
- **ARRAY**: `TYPE_NAME` ends with `[]`; the element type is the prefix, mapped recursively. Only
  element types the BE `DuckDbTypeHandler` decodes are allowed (scalars incl. LARGEINT/STRING);
  nested-of-nested and unmappable elements fail loud. `ARRAY<INT>`/`ARRAY<STRING>` are proven.
- **STRUCT/MAP/JSON → `STRING`** per the plan's v1 (§The dialect gap): "map DuckDB STRUCT/MAP/JSON
  to STRING/JSON on the FE". We use `STRING` (round-trippable text); first-class nested needs a
  BE handler arm (deferred).
- **INTERVAL → fail loud**: no faithful Doris type; doris-ducklake degrades it to STRING, but for
  duckbridge the value arrives through the JDBC handler (not a parquet reader), and DuckDB's
  interval string form isn't a Doris-castable literal — so we fail loud naming the column rather
  than risk a silently-wrong value. (Can be revisited as STRING if a round-trip is proven.)

### Types that FAIL LOUD (never silently skipped)
`UHUGEINT` (no 128-bit-unsigned Doris fit) and `INTERVAL` (no faithful mapping). The mapper throws
`DorisConnectorException` naming the column + DuckDB type. Everything else maps.

## TIMESTAMPTZ decision

**Decision (v1): `TIMESTAMPTZ` → `DATETIMEV2(6)`, naive typing over correct UTC instants — the same
choice doris-ducklake makes for its `timestamptz` columns with `enable.mapping.timestamp_tz=false`
(its default).**

Rationale (truthful zone story):
- quack-jdbc reports the type faithfully (`TIMESTAMP WITH TIME ZONE`), so we *could* declare a
  zone-aware Doris type. But the plan (§Timezone) gates **all tz-sensitive behavior OFF in v1**, and
  doris-ducklake's `DuckLakeTypeMapping` documents that the zone-aware Doris type (`TIMESTAMPTZ` →
  `ScalarType.createTimeStampTzType`) needs a **master/nightly BE** — the `Int64ToTimestampTz`
  converter is absent from 4.1.x, which errors `"DateTimeV2 => TimeStampTz"`. Our compose BE is
  branch-built but we do not depend on that converter for v1.
- DuckDB stores `TIMESTAMPTZ` as an **instant** (microseconds since epoch, UTC internally). Read as
  a naive `DATETIMEV2(6)`, the **wall-clock value is the correct UTC instant**; only the *zone-aware
  typing* is deferred. This is a documented, non-silent degradation — the column reads correct UTC
  values, and no tz-sensitive expression is pushed down (P3 gate), so there is no wrong-instant risk.
- We expose a catalog property `enable.mapping.timestamp_tz` (default `false`) mirroring the iceberg
  / ducklake knob, so a future zone-aware mapping is a config flip, not a code change. When `true`
  we map to `DATETIMEV2(6)` still (NOT a zone-aware type) but record the intent — flipping to a real
  zone-aware Doris type waits on the BE converter + P3.

This is truthful: we are NOT claiming zone-aware semantics. We claim correct UTC instants with naive
typing, tz pushdown gated off — stated plainly, never silently wrong.

## Driver quirks (for the record)

1. **`DATA_TYPE` is unreliable — always read `TYPE_NAME`.** ~40% of the surface (all wide/unsigned
   ints, sub-second/extended timestamps, UUID, INTERVAL, JSON, LIST, MAP, ENUM) reports
   `OTHER(1111)`; STRUCT reports `STRUCT(2002)`. Only common scalars get a real code.
2. **Unsigned ints report a *wider signed* code** (`UTINYINT`→SMALLINT code, `UINTEGER`→BIGINT
   code) — a sensible fallback, but `UBIGINT`/`UHUGEINT` overflow to `OTHER`. The `TYPE_NAME` is
   always exact, so we ignore the code entirely.
3. **`COLUMN_SIZE`/`DECIMAL_DIGITS` are only populated for the classic scalars** (ints, float,
   double, decimal). For DECIMAL they correctly carry precision/scale, but since `TYPE_NAME` already
   embeds `DECIMAL(p,s)` we parse that and treat size/digits as a cross-check only.
4. **Nested/parameterized types embed their full spec in `TYPE_NAME`** (`INTEGER[]`,
   `STRUCT(a INTEGER, b VARCHAR)`, `MAP(VARCHAR, INTEGER)`, `DECIMAL(38,10)`, `ENUM('a','b')`),
   which is exactly what a string type-parser needs.
5. **Base tables = `TABLE`; DuckDB internals = `SYSTEM VIEW`.** Catalogs `system`/`temp` and schema
   `information_schema`/`pg_catalog` must be filtered out of user-facing listing.
6. **Nullability is faithful** (`NOT NULL` → `columnNoNulls`).
7. **Unicode identifiers round-trip** through `getColumns` unchanged.

## Connection lifecycle & P2 note

The metadata plane opens a **short-lived quack-jdbc connection per resolution call** (list/resolve
at FE plan time) and closes it — the doris-ducklake pattern (plan-time catalog round-trips over a
JDBC connection). This is fine for correctness and keeps no long-lived state on the FE.

**P2 (pool pressure) note:** the FE metadata connections are separate from the BE scan connections.
Metadata resolution is low-frequency (catalog refresh / plan time) and one connection at a time, so
it does not stress Quack 1.5.4's fixed server-side pool. The pool question that gated trino-duckbridge
is about **per-split scan churn on the BE**, which is P2's scope (the BE's `JdbcJniScanner`
connection behavior) and unaffected by this FE metadata path. No pooling added on the FE side in v1;
if catalog refresh ever fans out, revisit.

## Open question: ATTACH / multi-catalog (deferred)

quack-jdbc exposes DuckDB's 3-level namespace, and a server can `ATTACH` multiple databases (each a
top-level catalog). v1 exposes only the connection's **main catalog** (discovered from the
connection, not hard-coded). Supporting multiple attached catalogs would need a mapping decision
(catalog→Doris-catalog vs catalog.schema→Doris-database) and is out of scope until a use case
appears. Recorded here so the flattening choice is a conscious v1 narrowing, not an oversight.
